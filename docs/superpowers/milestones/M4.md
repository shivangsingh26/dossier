# M4 — Job Inbox Page + Pipeline Run + SSE Progress (Scoped Stub)

> **Status:** Stub — flesh out into bite-sized tasks when M3 is complete. Say "expand M4".

**Spec:** [docs/superpowers/specs/2026-05-18-dossier-saas-frontend-design.md](../specs/2026-05-18-dossier-saas-frontend-design.md) §7 (credits), §8 (pipeline + worker), §9 (jobs + pipeline endpoints), §10 (job inbox layout).

**Goal:** The MVP-closing milestone. User clicks "Run discovery" → credits deduct → background worker runs `dossier_sdk.orchestrator.run_discovery` → SSE streams progress → jobs appear in inbox. User can mark status, take notes. Streamlit dashboard retired after this milestone.

**Architecture:** Frontend `/jobs` page with sidebar layout (KPI strip + tab filters + JobCard list + sticky JobDetail). Backend exposes `/jobs/*` (reads existing `data/{slug}/dossier.db`) and `/pipeline/run` (creates rows in `accounts.db`'s `pipeline_runs`, worker picks up). SSE endpoint streams worker progress. Credit gate enforces tier + balance atomically.

## Acceptance criteria (from spec §15 M4)

- User has 100 credits, lite tier
- Click "Run discovery" → confirm dialog "5 credits → 95 remaining" → confirm
- Credits drop from 100 → 95 immediately (optimistic UI, confirmed by backend)
- Progress overlay opens, shows "Queued → Running → Step 1/N → ..." via SSE
- Worker invokes `dossier_sdk.orchestrator.run_discovery` (which uses existing pipeline code)
- Run completes in 30–120s, overlay closes, jobs list refreshes
- New jobs appear in `/jobs`, sorted by score desc
- User clicks a card → detail pane opens with score reason + skill gaps + JD
- User clicks "⭐ Interested" → status updates in `dossier.db`, KPI strip updates
- User adds a note → persists, reload preserves note
- If worker fails (e.g. JSearch API down), credits refunded, error shown in overlay
- `pytest backend/tests/test_pipeline.py -v` covers credit deduct/refund/atomicity
- Streamlit `dashboard.py` is now redundant — recommend `git rm dashboard.py` at end of M4
