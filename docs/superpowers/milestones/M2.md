# M2 — Backend Skeleton + Clerk Webhook + accounts.db + Worker (Scoped Stub)

> **Status:** Stub — flesh out into bite-sized tasks when M1 is complete. Say "expand M2".

**Spec:** [docs/superpowers/specs/2026-05-18-dossier-saas-frontend-design.md](../specs/2026-05-18-dossier-saas-frontend-design.md) §6 (auth flow), §7 (credit/tier schema), §8 (pipeline + worker), §9 (FastAPI surface).

**Goal:** Stand up the FastAPI backend with Clerk JWT auth, `accounts.db` schema, signup webhook, seed script for existing 4 users, and a running (idle) worker process. Frontend's CreditPill becomes real.

**Architecture:** `backend/` Python 3.12 project with own venv. Depends on `dossier-sdk` as editable local path. FastAPI app on `:8000`. Pipeline worker = separate Python process polling `accounts.db` every 5s. JWT verified via `clerk-sdk-python`. CORS allows `localhost:3000`.

**Tech additions:** FastAPI, uvicorn, pydantic v2, clerk-sdk-python, sse-starlette, fasteners, sqlite3 (stdlib).

**Estimated time:** 6–10 hours.

Extracted from `run_dossier.py` during M2 (deferred from M0). Each returns a `RunResult` dataclass; `run_discovery` + `run_watchlist` return `(RunResult, list[dict])` because downstream stages need the jobs list. This is what FastAPI calls.

## Acceptance criteria (from spec §15 M2)

- `uvicorn dossier_api.main:app --reload` starts on `:8000`
- `GET /health` returns `{"ok": true}`
- `python -m dossier_api.workers.pipeline_worker` runs, logs `[idle] no queued runs`, sleeps, loops
- `python -m dossier_api.scripts.seed_existing_users` creates Clerk users + accounts.db rows for shivang/krishna/anushthan/sambhav (status=active, tier=max). Re-running is a no-op.
- Frontend signup flow → Clerk webhook fires → new row in accounts.db with status=pending, tier=lite, credits=100
- Frontend `/dashboard` for a pending user shows "Your account is pending review" screen
- Frontend `/dashboard` for active user shows CreditPill = real credit balance
- `pytest backend/tests/ -v` all green
